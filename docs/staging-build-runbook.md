# SampleFinder Staging Build & Distribute Runbook

How to build and distribute the side-by-side **SampleFinder (Staging)** app
(`com.samplefinder.app.staging`) against the staging Appwrite project. This is the detailed
companion to the **`release-staging`** skill (the build/gate/distribute shortcut) — it reproduces
the gitignored native staging edits below. Shared preflight / version bump / hard rules live in the
**`release`** core skill (prod build = **`release-prod`**, both at once = **`release-both`**).

- **Android status:** ✅ validated 2026-07-07 (APK built, verified, signed).
- **iOS status:** ✅ validated 2026-07-08 (archive built, verified, env gate passed).

---

## Hard rules (do not violate)

- **NEVER `expo prebuild`** (with/without `--clean`) and **NEVER `eas build`.** `android/`
  and `ios/` are gitignored; prebuild regenerates them from `app.json` + plugins and wipes
  every hand-edit — signing config, the Firebase notification-color fix, and the staging
  flavor/scheme this runbook adds. Edit the existing native dirs directly. (See the `release`
  skill for the full rationale.)
- **Always export `APP_VARIANT=staging`** for every staging build command (Metro bundling
  + gradle + xcodebuild). Without it you get a **production** app.
- **MANDATORY ENV GATE (never skip):** after any staging build, grep the JS bundle and assert
  the **staging** Appwrite project id is present and the **prod** id is absent. A staging app
  silently talking to prod is the worst failure mode here — see "Why the env gate exists".
- **Version numbers = prod's** (shared). No independent staging bumping.
- **Never** put a server Appwrite API key in the client.

---

## Effect on production builds (flavors changed the prod command + path)

Adding the Android product flavors (below) changes the **production** build too, even though prod is
otherwise untouched (same bundle id, env, Firebase, signing):

- Bare `./gradlew assembleRelease` now builds **both** flavors, and the prod APK moved into a flavor
  subdir. Build prod explicitly:
  ```bash
  cd android && unset APP_VARIANT && ./gradlew assembleProductionRelease
  # → android/app/build/outputs/apk/production/release/app-production-release.apk
  ```
- `unset APP_VARIANT` is the one real cross-contamination guard: the gradle **flavor** sets native
  identity (bundle id, google-services), while **`APP_VARIANT`** selects which `.env` is inlined into
  the JS bundle — two independent switches. A stale `APP_VARIANT=staging` would put staging backend
  values inside a prod-flavored app. A prod build should mirror the staging env gate (assert the
  **prod** id is present and the **staging** id absent in the bundle).

The workspace `release` skill (the canonical prod procedure) was updated to match; this section is the
git-tracked record of that change.

## Why the env gate exists (the env-precedence trap — already fixed, keep the gate)

`@expo/env` auto-loads `.env` (prod) into `process.env` at Expo CLI startup. `react-native-dotenv`
merges its `path` file (`.env.staging`) with a snapshot of `process.env` where **`process.env`
wins**. Result: staging builds silently inlined the **prod** Appwrite project id even though
`babel.config.js` correctly selected `.env.staging`.

**Fix (already in `babel.config.js`, committed 1efbfbc):** for the staging variant only, re-load
`.env.staging` into `process.env` with `override: true` before react-native-dotenv reads it:

```js
if (variant === 'staging') {
  require('dotenv').config({ path: require('path').join(__dirname, envFile), override: true });
}
```

This is why the build works — but keep the grep gate anyway, because this class of bug is silent
and only the bundle contents prove which backend the app will hit.

## Preflight (do ALL before building)

1. **Branch:** be on `feature/staging-environment` (or wherever staging lives). `git rev-parse --abbrev-ref HEAD`.
2. **Release skill Step 0** (verbatim): keystore at repo-root `release.keystore`, release `signingConfig`
   in `android/app/build.gradle`, `android/local.properties`, disk (≥ ~6 GB; iOS archive needs it —
   `rm -rf ~/.gradle/caches` if starved), pods (`npx pod-install ios` if building iOS).
3. **`.env.staging` present** (gitignored) with the 3 deltas vs prod:
   `APPWRITE_PROJECT_ID=6a0ad92e0001d5e515ce`, `APPWRITE_PLATFORM=com.samplefinder.app.staging`,
   `DEEP_LINK_SCHEME=samplefinderstaging`. All other keys identical to prod `.env`.
4. **Firebase config files present at repo root:** `google-services.staging.json`,
   `GoogleService-Info.staging.plist` (both carry `com.samplefinder.app.staging`).
5. **Native staging edits applied** (below). If `android/`/`ios/` were just regenerated by a bootstrap
   prebuild, re-apply them from this runbook.
6. **Clear Metro cache** before building (belt-and-suspenders; the env fix is the real safeguard):
   `rm -rf "$TMPDIR"/metro-* "$TMPDIR"/metro-cache node_modules/.cache .expo`

## Native edits — Android (gitignored; reapply after any bootstrap prebuild) — ✅ validated

**`android/app/build.gradle`** — inside `android { }`, right after `defaultConfig { }`:
```gradle
    flavorDimensions "env"
    productFlavors {
        production {
            dimension "env"
            // applicationId stays com.samplefinder.app (from defaultConfig)
        }
        staging {
            dimension "env"
            applicationIdSuffix ".staging"
        }
    }
```
The `release` buildType's `signingConfig` (release skill Step 0b) applies to both flavors — same keystore.

**`android/app/src/main/AndroidManifest.xml`** — REMOVE the `samplefinder.com` autoVerify App Link
`<intent-filter>` from `.MainActivity` (keep the MAIN/LAUNCHER filter). It moves to the production flavor.

**`android/app/src/production/AndroidManifest.xml`** (new — App Link belongs to prod only):
```xml
<manifest xmlns:android="http://schemas.android.com/apk/res/android">
  <application>
    <activity android:name=".MainActivity">
      <intent-filter android:autoVerify="true">
        <action android:name="android.intent.action.VIEW" />
        <category android:name="android.intent.category.DEFAULT" />
        <category android:name="android.intent.category.BROWSABLE" />
        <data android:scheme="https" android:host="samplefinder.com" android:pathPrefix="/referral/" />
      </intent-filter>
    </activity>
  </application>
</manifest>
```

**`android/app/src/staging/AndroidManifest.xml`** (new — custom scheme only, no autoVerify):
```xml
<manifest xmlns:android="http://schemas.android.com/apk/res/android">
  <application>
    <activity android:name=".MainActivity">
      <intent-filter>
        <action android:name="android.intent.action.VIEW" />
        <category android:name="android.intent.category.DEFAULT" />
        <category android:name="android.intent.category.BROWSABLE" />
        <data android:scheme="samplefinderstaging" />
      </intent-filter>
    </activity>
  </application>
</manifest>
```

**`android/app/src/staging/res/values/strings.xml`** (new):
```xml
<resources>
    <string name="app_name">SampleFinder (Staging)</string>
</resources>
```
(Leave `src/main/res/values/strings.xml` `app_name` = `SampleFinder` for the production flavor.)

**`android/app/src/staging/google-services.json`** — copy of the staging Firebase config:
```bash
cp google-services.staging.json android/app/src/staging/google-services.json
```

## Build & verify — Android — ✅ validated

```bash
cd samplefinder-app
rm -rf "$TMPDIR"/metro-* "$TMPDIR"/metro-cache node_modules/.cache .expo    # preflight step 6
APP_VARIANT=staging ./android/gradlew -p android :app:assembleStagingRelease --console=plain
# if you just cleared ~/.gradle/caches: ./android/gradlew --stop first (stale daemon)
```
Output: `android/app/build/outputs/apk/staging/release/app-staging-release.apk` (~94 MB).

**Verify (all must pass):**
```bash
APK=android/app/build/outputs/apk/staging/release/app-staging-release.apk
BT="$HOME/Library/Android/sdk/build-tools/36.0.0"
"$BT/aapt2" dump badging "$APK" | grep -E "^package:|application-label:'"
# expect: package name='com.samplefinder.app.staging' ...  application-label:'SampleFinder (Staging)'
"$BT/apksigner" verify --print-certs "$APK" | grep -i "DN:"
# expect: CN=SampleFinder, ... Polaris Brand Promotions   (NOT Android Debug)

# --- MANDATORY ENV GATE ---
SCRATCH=$(mktemp -d); unzip -o -q "$APK" assets/index.android.bundle -d "$SCRATCH"
B="$SCRATCH/assets/index.android.bundle"
grep -a -c "6a0ad92e0001d5e515ce" "$B"   # staging id — MUST be >= 1
grep -a -c "691d4a54003b21bf0136" "$B"   # prod id    — MUST be 0
rm -rf "$SCRATCH"
```
`grep -c` on the raw (Hermes) stream can print nothing; extract to a file and grep with `-a` as above.

**Distribute (Android):** the APK (~94 MB) exceeds the Appwrite `files` bucket cap (50 MB) — do NOT
upload it there. Use a Google Drive link:
```bash
cp "$APK" "$HOME/Desktop/SampleFinder-Staging-<ver>-build<n>.apk"
```
Upload to Drive → "Anyone with the link". Handoff: `Android (Staging): <ver> build <n> — <drive link>`.

## Native edits — iOS (gitignored, via Xcode + Podfile) — ✅ validated 2026-07-08

Open `ios/SampleFinder.xcworkspace`.

1. **Duplicate** the `Release` build configuration → name it **`Release-Staging`** (select the
   **project** → Info → Configurations → ＋ → Duplicate "Release").
2. **Target SampleFinder → Build Settings, `Release-Staging` column only:**
   - `PRODUCT_BUNDLE_IDENTIFIER` = `com.samplefinder.app.staging`
   - `CODE_SIGN_ENTITLEMENTS` = `SampleFinder/SampleFinderStaging.entitlements`
   - `PRODUCT_NAME` = **leave `$(TARGET_NAME)`** — do NOT put the display name here; a space in
     `PRODUCT_NAME` flows into `WRAPPER_NAME`/executable paths and can break RN build phases.
3. **Display name** (home-screen label `SampleFinder Staging`). The shared `Info.plist` hardcodes
   `CFBundleDisplayName`, so make it a per-config variable:
   - Set `Info.plist` `CFBundleDisplayName` = `$(APP_DISPLAY_NAME)`.
   - Add a **User-Defined** build setting `APP_DISPLAY_NAME`: base value `SampleFinder` (keeps
     Debug/Release/**prod** correct), Release-Staging override `SampleFinder Staging`.
4. **`ios/SampleFinder/SampleFinderStaging.entitlements`** (new — push only; **mirror prod's
   `aps-environment=development`** — the distribution export re-signs to production APNs on upload;
   NO associated domains):
   ```xml
   <?xml version="1.0" encoding="UTF-8"?>
   <!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
   <plist version="1.0">
   <dict>
     <key>aps-environment</key>
     <string>development</string>
   </dict>
   </plist>
   ```
5. **Staging GoogleService plist:** `cp GoogleService-Info.staging.plist ios/SampleFinder/`. It does
   NOT need to be added to the Xcode project — the run-script (next) copies it by path.
6. **Run Script build phase** (target → Build Phases → ＋ → New Run Script Phase → drag it AFTER
   "Copy Bundle Resources"; uncheck "Based on dependency analysis"):
   ```bash
   if [ "${CONFIGURATION}" = "Release-Staging" ]; then
     cp "${SRCROOT}/SampleFinder/GoogleService-Info.staging.plist" \
        "${BUILT_PRODUCTS_DIR}/${WRAPPER_NAME}/GoogleService-Info.plist"
   fi
   ```
7. **URL scheme:** add `samplefinderstaging` to `Info.plist` `CFBundleURLTypes` (a second entry
   alongside prod's `com.samplefinder.app`; prod ignores it):
   ```bash
   /usr/libexec/PlistBuddy -c "Add :CFBundleURLTypes:1:CFBundleURLSchemes array" ios/SampleFinder/Info.plist
   /usr/libexec/PlistBuddy -c "Add :CFBundleURLTypes:1:CFBundleURLSchemes:0 string samplefinderstaging" ios/SampleFinder/Info.plist
   ```
8. **Shared scheme** `SampleFinder-Staging` (Product → Scheme → Manage Schemes → duplicate
   `SampleFinder`) → Edit Scheme → all actions (Run/Test/Profile/Analyze/Archive) = `Release-Staging`
   → mark **Shared**.
9. **⚠️ CocoaPods — REQUIRED, the easy-to-miss step:** a new Xcode build configuration is invisible to
   CocoaPods until you map it and re-install, or the archive fails with
   `module map file '…/Release-Staging-iphoneos/EXApplication.modulemap' not found` (every Expo pod).
   In `ios/Podfile`, add before the `target` block:
   ```ruby
   project 'SampleFinder',
     'Debug' => :debug,
     'Release' => :release,
     'Release-Staging' => :release
   ```
   then `npx pod-install ios`. Verify `Pods/Target Support Files/Pods-SampleFinder/Pods-SampleFinder.release-staging.xcconfig`
   exists and the app target's Release-Staging base config points at it.

## Build & verify — iOS — ✅ validated 2026-07-08

```bash
cd samplefinder-app
# If you just changed the Pods project (step 9) or a prior archive failed, clear DerivedData first:
rm -rf "$HOME/Library/Developer/Xcode/DerivedData/SampleFinder-"*
ARCHIVE_DIR="$HOME/Library/Developer/Xcode/Archives/$(date +%Y-%m-%d)"; mkdir -p "$ARCHIVE_DIR"
V=$(node -p "require('./app.json').expo.version"); B=$(node -p "require('./app.json').expo.ios.buildNumber")
ARCHIVE_PATH="$ARCHIVE_DIR/SampleFinder-Staging-$V-build$B.xcarchive"
APP_VARIANT=staging xcodebuild -workspace ios/SampleFinder.xcworkspace \
  -scheme SampleFinder-Staging -configuration Release-Staging \
  -destination 'generic/platform=iOS' \
  -archivePath "$ARCHIVE_PATH" -allowProvisioningUpdates archive
open "$ARCHIVE_PATH"   # Organizer → Distribute → App Store Connect → TestFlight
```
Run in the background (RN archives are slow) and grep the log for `** ARCHIVE SUCCEEDED **`.

**Verify (all must pass) on `$ARCHIVE_PATH/Products/Applications/SampleFinder.app`:**
```bash
APP="$ARCHIVE_PATH/Products/Applications/SampleFinder.app"
/usr/libexec/PlistBuddy -c "Print :CFBundleIdentifier" "$APP/Info.plist"       # com.samplefinder.app.staging
/usr/libexec/PlistBuddy -c "Print :CFBundleDisplayName" "$APP/Info.plist"      # SampleFinder Staging
/usr/libexec/PlistBuddy -c 'Print :BUNDLE_ID' "$APP/GoogleService-Info.plist"  # com.samplefinder.app.staging (plist swap worked)
codesign -d --entitlements :- "$APP" 2>/dev/null | grep -iE "aps-environment|associated-domains"  # aps dev; NO associated-domains
# --- MANDATORY ENV GATE ---
grep -a -c "6a0ad92e0001d5e515ce" "$APP/main.jsbundle"   # staging id — MUST be >= 1
grep -a -c "691d4a54003b21bf0136" "$APP/main.jsbundle"   # prod id    — MUST be 0
```

**Distribute (iOS):** Organizer → Distribute App → App Store Connect → TestFlight. Uploads to the
**staging** App Store Connect record (`com.samplefinder.app.staging`). Handoff: `iOS (Staging): TestFlight (<ver> build <n>)`.

## Dev runtime (no native build)

```bash
cd samplefinder-app && npm run start:staging    # APP_VARIANT=staging expo start
```
The installed dev client uses prod native config (prod Firebase); Appwrite connectivity + `@env`
resolve to staging. Good for JS/config QA; real staging push must be tested on the signed staging APK.
